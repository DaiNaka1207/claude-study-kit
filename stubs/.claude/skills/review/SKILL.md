# /review — 提出物レビュー・評価保存

提出物: **$ARGUMENTS**

## あなたのタスク

### Step 1: タスク内容の確認

`.study/_session/current_session.md` からトピックとタスクを特定する。
対応する `TASK.md` を読んで以下を確認する：
- タスクの目的
- 完了基準（チェックリスト）
- ヒントログ（ヒント要求回数）

### Step 2: 提出物の評価

以下の4観点でレビューする。**コードの正しさだけでなく思考プロセスを評価すること。**

1. **詰まり方の質** — 構文エラーによる詰まりか、概念的な誤解によるものか
2. **ヒント要求回数** — 0回＝高理解、1〜2回＝標準、3回以上＝サポート必要
3. **自分の言葉での説明** — 「なぜこう書いたか」を説明できているか
4. **完了基準の達成率** — チェックリストの何割を満たしているか

### Step 3: 理解度レベルの判定

| レベル | 判定基準 |
|--------|----------|
| **A（深い理解）** | 自分の言葉で説明できる。ヒントなし。応用への質問ができる |
| **B（基本的な理解）** | 基本実装完了。多少ヒントあり。「なぜ」を完全には説明できない |
| **C（要復習）** | 完了基準は満たしたが試行錯誤が多い。同概念を別角度で再練習必要 |
| **D（再挑戦推奨）** | 完了基準未達、または根本的な誤解あり。前提知識の確認が必要 |

### Step 4: レビューファイルの作成

`.study/topics/{topic}/00_reviews/YYYYMMDD_NN_task.md` を新規作成する。
`.study/_templates/REVIEW_template.md` を参照すること。

### Step 5: PROGRESS.md の更新

`.study/topics/{topic}/PROGRESS.md` の評価テーブルに1行追加する：

```
| {日付} | {タスク名} | {レベル} | {ヒント回数} | {コメント} |
```

### Step 6: current_session.md の更新

```
Status: Review Complete
Last Review: {日付}
Last Level: {A/B/C/D}
```

### Step 7: フィードバックと次のアクション提案

フィードバックは**質問を交えて**行う（「正解のコード」は示さない）。
レベルに応じて次のステップを提案し、`/next` を実行するよう促す。

---

> ⚠️ フィードバック時も「正解のコード」は絶対に提示しないこと。
> 改善点は「この部分、なぜこう書きましたか？」のように問いかける形で伝える。
